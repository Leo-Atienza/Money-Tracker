# Money Tracker — Adversarial Audit Findings (2026-06-26)

10-dimension adversarial Workflow audit (86 agents); every finding refute-checked before inclusion. **62 confirmed**, 14 refuted. Severity: {'critical': 0, 'high': 2, 'medium': 19, 'low': 41}.

Already fixed in the 2026-06-26 session (NOT in the list below): systemic black-background regression (global OrganicBlobBackground via MaterialApp.builder), H1 PIN re-lock-on-resume, and a wall-clock-rot test.


## HIGH (2)

### H1. App never re-locks: PIN unlock screen is checked only once at startup, not on resume from background
- **Dimension:** state-lifecycle | **Confidence:** 0.92 | **Verdict:** real
- **Location:** `lib/main.dart:463-476 (didChangeAppLifecycleState) vs 400-403/479-499 (_checkPinLock)`
- **Why:** On background (paused), _handlePaused() calls appState.lock() (and the 3-minute inactivity Timer also calls lock()), setting _isLocked=true. But the resumed lifecycle handler in MainNavigationScreen only re-checks notifications — it never re-invokes _checkPinLock(). MyApp's resumed handler only calls loadData(). So after the app is backgrounded and re-foregrounded, the PinUnlockScreen is never re-presented even though state is locked, and the user sees all financial data without re-entering the PIN. This defeats the entire auto-lock-on-background feature (and the Phase 6.x FLAG_SECURE work that assumes a real lock gate).
- **Fix:** In MainNavigationScreen.didChangeAppLifecycleState, in the AppLifecycleState.resumed branch (lib/main.dart:470-475), also schedule a post-frame `_checkPinLock()` next to the existing `_checkPendingNotification()` (both mounted-guarded). `_checkPinLock` already early-returns when `!appState.isLocked`, so it is a no-op when unlocked. Add a re-entrancy guard: a `bool _pinRouteActive` set true immediately before `Navigator.push` and cleared in a `finally` after the await (lib/main.dart:484-490), and skip the push when `_pinRouteActive` is already true, so a second resume while PinUnlockScreen is on the stack doesn't push a duplicate.

### H2. App PIN stored as single-iteration SHA-256 — trivially brute-forceable offline
- **Dimension:** security | **Confidence:** 0.9 | **Verdict:** real
- **Location:** `lib/utils/pin_security_helper.dart:230-234`
- **Why:** The unlock secret is a 4-6 digit PIN (10^4-10^6 = at most 1M candidates) hashed with ONE round of unsalted-stretching SHA-256. The salt only defeats precomputed rainbow tables; it does nothing against a targeted search. If the Keystore-backed value is ever extracted (rooted device, Keystore-extraction CVE, backup of the secure store, or the `encryptedSharedPreferences` fallback file), an attacker recovers the PIN in well under a second on commodity hardware — billions of SHA-256/s on a GPU vs. a 1M keyspace. Notably the project already depends on `package:cryptography` and uses PBKDF2 with 100k iterations for backup encryption (backup_crypto.dart:35), so the slow-KDF machinery is right there but is not used for the PIN.
- **Fix:** In _hashPinWithSalt, replace the single sha256.convert with the PBKDF2 derivation already used by BackupCrypto: derive a 256-bit key via Pbkdf2(macAlgorithm: Hmac.sha256(), iterations: 100000, bits: 256) using the base64-decoded per-PIN salt as the PBKDF2 salt (nonce), then base64/hex-encode the derived key for storage. Keep _constantTimeEquals for the compare. Because the plaintext PIN is only available at verify time, add a one-time re-derivation in verifyPin: when the stored hash is detected as the legacy single-SHA-256 format and the input matches it (or the storedSalt == null legacy branch), call setPin(pin) again on success to transparently upgrade to the PBKDF2 hash. Note _hashPinWithSalt is currently synchronous while Pbkdf2.deriveKey is async, so the method (and its setPin/verifyPin callers, already async) will need to become Future-returning.


## MEDIUM (19)

### M3. addPayment silently records money the user never paid (auto-round-up under 10 cents)
- **Dimension:** money-math | **Confidence:** 0.9 | **Verdict:** partial
- **Location:** `lib/providers/app_state.dart:802-812`
- **Why:** When a partial payment leaves under 10 cents remaining, the code overwrites amountPaid with the full expense amount, marking the bill fully paid and inflating amountPaid by up to 9 cents the user never paid. This corrupts getAvailableIncomeForMonth (which subtracts amountPaid, app_state.dart:2210), totalPaid (app_state.dart:2205), and CSV/export figures. A 9-cent fabrication per bill is a real accounting error in a money tracker, and it is invisible to the user. The validator even pre-blesses small overshoots (add_payment_dialog.dart:394 `if (amount > remaining + 0.001)`).
- **Fix:** Remove the auto-round-up branch (app_state.dart:807-809) entirely. Keep only the overpayment cap (line 804) and the exact-amount else branch, so finalAmountPaid = newAmountPaidDecimal records exactly what was paid. Users who want to clear a bill already have the 'Pay All' button (add_payment_dialog.dart:404) which sets the field to the full remaining.

### M4. Crash-log PII redaction misses free-text descriptions and symbol-less amounts that DB errors echo
- **Dimension:** security | **Confidence:** 0.78 | **Verdict:** partial
- **Location:** `lib/utils/crash_log.dart:288-330`
- **Why:** The crash log is exportable/shareable (class doc, lines 22-24) so its contents are an exfiltration surface. The redactor masks only currency-*symbol*-tagged numbers, but this app stores money as `package:decimal` and surfaces it without a symbol in many code paths; a SQLite/validation exception such as `CHECK constraint failed: amount=1234.56` or a wrapped error carrying `description='Dinner with Jane at 123 Main St'` is logged verbatim because (a) `1234.56` has no leading currency glyph and (b) merchant/note free text matches none of the four regexes. The redactor's own doc claims to target the app's 'core data is money', yet the dominant in-app representation (symbol-less Decimal + user descriptions/category names) slips through.
- **Fix:** In CrashLog.record (or a pre-processing step before _formatRecord), special-case sqflite DatabaseException: log only the constraint/error message and SQL template, and strip the trailing `args [...]` segment that echoes bound parameter values, e.g. regex-remove ` args \[.*?\]$` from the error string. This is data-minimization at the source. Do NOT add a broad symbol-less decimal regex to redactPii — `\d+(?:\.\d+)?` would over-mask timestamps, line numbers, and IDs in every stack trace. Optionally also widen _currencyRe to catch a few known formatted-balance shapes, but the args-stripping is the load-bearing fix.

### M5. PDF/CSV export runs full doc generation + zlib compression synchronously on the UI isolate (no compute())
- **Dimension:** performance | **Confidence:** 0.8 | **Verdict:** partial
- **Location:** `lib/utils/pdf_exporter.dart:355 (also 633, 877; csv_exporter.dart 131/244/338)`
- **Why:** exportExpensesToPdf/exportMonthlySummaryToPdf build a pw.Table with one TableRow per transaction (up to _maxTotalResults=1000 rows from history) and then call pdf.save(), which does layout + Deflate compression entirely on the main isolate. For a few-hundred-to-1000-row report this is a multi-hundred-millisecond block — the UI freezes and the export spinner itself can't animate. backup_helper.dart already proves the pattern is available (it wraps serialization in compute() at lines 482/622/1051); the exporters were never moved off-thread.
- **Fix:** Move the CPU-bound serialization off the UI isolate via compute(), mirroring backup_helper.dart's existing pattern. For PDF: build the pw.Document and call pdf.save() inside a top-level isolate function that returns Uint8List, then writeAsBytes on the main isolate (pdf_exporter.dart:355, 633, 877). For CSV: run the StringBuffer-building loop in a top-level function via compute() returning the final String, then writeAsString (csv_exporter.dart:131, 244, 338). Pass the already-materialized plain List<Expense>/List<Income> plus the scalar params (currency symbol/code, separator, dates) — these are isolate-transferable. Note the data source is the unbounded getAll...ForBackup() fetch (export_data_screen.dart:528-529), not the history 1000-row cap, so size the concern accordingly.

### M6. Analytics BudgetProgress is O(budgets × allExpenses + recurring) on every rebuild via getSpentForCategory in a .map() loop
- **Dimension:** performance | **Confidence:** 0.82 | **Verdict:** partial

### M7. History grouped list builds an entire day/category group's items eagerly inside one ListView item (non-virtualized)
- **Dimension:** performance | **Confidence:** 0.85 | **Verdict:** partial

### M8. Every history tile is wrapped in StaggeredListItem, each spawning its own AnimationController + Future.delayed
- **Dimension:** performance | **Confidence:** 0.8 | **Verdict:** partial
- **Location:** `lib/screens/history/history_list.dart:148-158`
